# 达人撮合平台 MVP 实现 TODO

本文档基于 `docs/prd.md` 与 `docs/tdd.md` 整理，用于把商家-达人撮合小程序从当前原型推进到可交付 MVP。每一项完成后勾选对应复选框。

## 0. 实施原则与范围锁定

- [x] 明确 MVP 只实现两条主路径：商家“注册/登录 -> 完善信息 -> 发布产品 -> 筛选达人 -> 解锁联系方式”，达人“注册/登录 -> 完善信息 -> 设置公开 -> 浏览产品”。
- [x] 明确暂不实现：复杂推荐算法、IM 聊天、支付分账、AI 匹配、缓存架构、对象存储。
- [x] 保留当前微信小程序原生框架，不切换 Taro，避免重构成本扩大。
- [x] 后端先按当前 Spring Boot 项目继续推进，再逐步替换内存数据为 MySQL 持久化。
- [x] 统一接口命名规范：前端调用路径与后端 Controller 保持一致；如继续使用当前 `/api/products` 风格，则在 TODO 后续步骤中同步更新前端；如改成 TDD 的 `/api/product/list` 风格，则一次性改 Controller、Service、前端请求层。

## 12. 前端视觉设计约束

这一节就是约束 AI 生成页面“不要难看”的地方。后续让 AI 生成或修改页面时，先要求它遵守本节，再让它写 WXML/WXSS。

- [x] 在 `app.wxss` 中维护设计令牌：背景色、主色、文字色、边框色、间距、圆角、阴影、字号。
- [x] 页面整体风格定位为“专业、清爽、可信赖的撮合工作台”，不要做夸张营销页、过度渐变、大面积插画、花哨动效。
- [x] 主色建议使用克制蓝色或蓝绿色，辅以少量绿色表达公开/成功状态、少量红色表达风险/下架状态；避免整页只有一种颜色。
- [x] 卡片圆角保持 `8rpx` 左右，不做大圆角气泡风。
- [x] 页面背景使用浅灰，内容区使用白色卡片或分组，不要层层嵌套卡片。
- [x] 标题层级清晰：页面标题、区块标题、卡片标题、说明文字字号要有差异，不要所有文字一样大。
- [x] 表单控件高度、内边距、边框、提示文字统一，输入项之间保持稳定间距。
- [x] 按钮分主次：主操作用 `.primary-btn`，次操作用 `.ghost-btn`，危险操作用 `.danger-btn`。
- [x] 筛选条件优先用分组输入、选择器、开关、标签，不要堆一排普通文本按钮。
- [x] 列表卡片要突出关键字段：名称、领域/平台标签、粉丝区间、预算/报价、状态；次要信息放弱色。
- [x] 所有页面必须检查空状态、加载状态、错误状态，不能只做有数据时的好看页面。
- [x] 文案要短，直接服务操作，不要在业务页面放大量功能介绍。
- [x] 移动端优先，保证 320px 宽度下文字不溢出、按钮不挤压、卡片不横向滚动。
- [ ] 页面生成后必须人工检查 WXML 结构与 WXSS：无文本重叠、无按钮文字溢出、无无意义装饰元素。

可直接复制给 AI 的页面生成约束：

```text
请按现有微信小程序原生结构生成页面，必须复用 app.wxss 中的 page、section、card、row、title、subtitle、tag、muted、primary-btn、ghost-btn、danger-btn、field、label、input、textarea、grid、empty 等基础类。页面风格要专业、清爽、可信赖，像一个商家/达人撮合工作台，不要做营销落地页，不要大面积渐变、夸张插画、过度圆角、卡片套卡片。每个页面都要包含加载、空状态、错误状态或必要的表单校验。移动端优先，按钮和文字不得溢出。
```
